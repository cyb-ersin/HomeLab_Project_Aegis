## Overview

This chapter documents the deployment of a Suricata IDS + Wazuh SIEM stack on a physical homelab environment, followed by two controlled attack simulations: a network reconnaissance scan and an SSH brute force attack. All attacks were detected, logged, and correlated with MITRE ATT&CK techniques.

## Phase 3 — Attack Simulation #1: Network Reconnaissance

### Attack

Nmap aggressive scan launched from MacBook against aegis-sentinel:

```bash
sudo nmap -sS -A -T4 172.20.10.6
```

**Flags:**
- `-sS` — SYN stealth scan (half-open, raw packets)
- `-A` — aggressive: OS detection + service version + scripts + traceroute
- `-T4` — fast timing template

**Results:**
```
PORT   STATE  SERVICE  VERSION
22/tcp open   ssh      OpenSSH 9.6p1 Ubuntu
OS:    Linux 4.15-5.19
MAC:   CA:24:5F:0E:0F:FD
```

### Detection

Suricata detected anomalous ICMP probes generated by nmap's OS fingerprinting phase.

![Wazuh Events — nmap Suricata alerts](screenshots/06_nmap_suricata_alert.png)

**Alert breakdown:**

|:------|:------|
| `rule.description` | Suricata: Alert - SURICATA ICMPv4 unknown code |
| `rule.id` | 86601 |

Hydra SSH brute force launched from physical Kali laptop against aegis-sentinel:


## Key Findings

**Finding 1 — Weak SSH password**
User `aegis-siem` had password `qwerty` — position ~491 in rockyou.txt. Cracked in under 2 minutes with `-t 4`. Mitigation: enforce strong password policy or switch to SSH key authentication.

**Finding 2 — Root SSH login blocked**
`PermitRootLogin prohibit-password` was correctly configured — root brute force attempts failed. This is the expected hardened default.

**Finding 3 — IDS in passive mode**
Suricata runs in IDS mode (`allowed` action on all alerts). Attacks were detected but not blocked. Mitigation: configure active response in Wazuh or switch Suricata to IPS mode.

**Finding 4 — Compliance violations auto-flagged**
Every brute force alert was automatically mapped to GDPR, PCI DSS, HIPAA, and NIST 800-53 frameworks by Wazuh — demonstrating automated compliance monitoring capability.

**Finding 5 — Dynamic IP instability**
Running the lab on an iPhone hotspot caused frequent IP changes, breaking Wazuh Agent → Manager connectivity. Mitigation: use a stable router or configure static IPs.

---

## Lessons Learned

- Suricata alone sees the network; Wazuh alone sees the system. Together they provide full SOC visibility.
- Wazuh's rule escalation (level 5 → 10 → 12) mirrors how real SOC triage works: individual events are noise, patterns are signals.
- MITRE ATT&CK mapping happens automatically — every alert is pre-tagged, enabling instant threat classification.
- `rockyou.txt` makes short work of common passwords. Any password in the top 10,000 is effectively no password.
